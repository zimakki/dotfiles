Should this repo adopt mise bootstrap?

Yes—as the coordinator. Fix four setup problems first, and keep the Brewfile as the package list.

Short answer

yes · selective adoption

Let mise run the setup sequence, but do not ask it to own everything.

Think of mise as the conductor: it tells the existing parts when to run and adds useful dry-runs and status checks.

mise bootstrap

Coordinates setup, installs mise tools, applies simple links and defaults, and reports drift.

BrewFile

Remains the one list of Homebrew formulae, casks, fonts, and the Inkwell tap.

zsh files

Continue to own shell activation and PATH behavior because the whole files are already managed.

tiny scripts

Handle the few exceptions that mise cannot express cleanly.

Fix these four things first

These are mostly problems in the current fresh-machine instructions—not disadvantages created by mise.

01Upgrade mise

The installed version is 2026.7.1. Bootstrap became stable in 2026.7.4. Upgrade first and require that minimum version.

02Make the config visible

The implemented discoverable repo-level mise.toml exposes tools before mise install and links itself globally.

03Correct the install order

Install prerequisites, load config and build settings, install packages, then install tools, links, and defaults—in that order.

04Repair shell startup

Replace the old pasted activation code, stop hardcoding one mise path, and ensure mise shims still win in login shells.

Why this matters: as written today, a new machine reaches mise install before it can see the tracked tool list.

What are the disadvantages?

It is powerful

Bootstrap can install packages, change files and preferences, and run hooks.

Run only trusted config and use --dry-run first. Avoid blind --yes / --force runs.
It is newly stable

The feature only left experimental mode in 2026.7.4, so details may still evolve.

Set min_version and start with a small pilot.
Homebrew support is incomplete

Our font casks and Inkwell tap are not fully representable, and cask import/prune is missing.

Keep the BrewFile intact; do not split package ownership.
Not everything becomes reversible

Removing config does not automatically undo old links/defaults, and a few custom steps remain.

Keep backups, snapshot defaults, and retain verify_setup.sh.

None of these block adoption. They only argue for using mise as the coordinator instead of replacing every existing mechanism.

What should mise own?

Move into mise

7 tools, 19 static symlinks (including zprofile), 12 typed macOS defaults, dry-run, and drift status.

Keep in BrewFile

All 64 package/app entries, including three font casks and the Inkwell tap.

Keep in zsh files

Interactive activation, non-interactive shims, and login-shell PATH precedence.

Keep as small exceptions

One dynamic Lazygit link, one -currentHost preference, skill sync, and GUI follow-up.

A safe adoption plan

  1. Repair the current ordering and shell issues.
  2. Upgrade mise to 2026.7.4+ and declare min_version.
  3. Add one discoverable repo-level mise.toml.
  4. Pilot only the high-fit pieces: tools, static links, and typed defaults.
  5. Keep the BrewFile and small exception scripts. Revisit package migration later.

Verdict

selective adoption

Use mise bootstrap as the future orchestrator, but do not replace the Brewfile wholesale.

The upstream model is a strong fit for our static symlinks, typed macOS defaults, tool installation, dry-runs, and drift reporting. It is a weak fit for our complete Homebrew surface today: mise’s built-in cask support excludes font artifacts, cask import/prune is absent, and our Inkwell tap does not publish the API metadata mise expects.

There is a more urgent result than the migration choice: the documented fresh-machine sequence is currently broken in several places. Repair those first, then pilot the declarative pieces behind a discoverable repo-level config.

What the upstream page proposes

The reviewed page describes mise bootstrap as a single convergence pass over packages, repositories, dotfiles, shell activation, macOS defaults, user services, login shell, mise tools, and a final custom task. Declarative steps skip state that already matches; the custom bootstrap task runs every time and must be idempotent. 

$ mise bootstrap --dry-run
$ mise bootstrap --yes
$ mise bootstrap status --missing
$ mise bootstrap --only dotfiles,tools
$ mise bootstrap --skip tools,task

Status correction: bootstrap and dotfiles graduated from experimental in mise 2026.7.4 on 2026-07-09. Our installed 2026.7.1 binary predates that release and still refuses the commands unless MISE_EXPERIMENTAL=1. Any adoption should upgrade first and declare min_version = "2026.7.4" or newer.

Bootstrap assumes mise itself is already installed and its config is discoverable. The official macOS installation recommendation remains Homebrew, with mise.run as the alternative; bootstrap is not a zero-dependency installer for mise itself.

What we actually have

Today mise owns only the seven entries under [tools] in mise.toml. The rest of machine setup is split across a Brewfile, a custom symlink installer, a macOS defaults script, and prose commands in the migration guide. 

packages       BrewFile                           # 40 formulae + 24 casks
tools          mise.toml                          # 7 entries
dotfiles       mise.toml + exception script       # 20 targets
shell          zshenv + pasted code in zshrc       # shims + interactive hook
defaults       macos_defaults.sh                  # 13 writes
orchestration  MIGRATION.md                       # manual six-step runbook

With the experimental flag temporarily enabled, live mise bootstrap status --json reports empty packages, repos, dotfiles, shell activation, defaults, services, and login shell; only the seven tools appear. That is the cleanest possible statement of the gap between the new page and this repo.

Fresh-machine blockers found during comparison

These are correctness problems in the current path, independent of whether we adopt the new feature.

1. The preflight requires the tools that the later steps are meant to install

phase2_preflight.sh blocks when Homebrew or mise is absent, yet MIGRATION.md installs mise only later through Brew Bundle. The stated prerequisites never install either one.

2. mise install runs before its tracked config exists

The runbook calls mise install, then runs the symlink installer that finally creates ~/.config/mise/config.toml. The previous mise_config.toml was not an auto-discovered local filename, a genuinely fresh home has no requested tools at install time.

3. Erlang’s build setting also arrives too late

KERL_BUILD_DOCS=yes lives in the zshrc that is linked after Erlang installation. The file itself says the setting must exist before installation. The build dependencies are present, but the switch that uses them is not guaranteed.

Result: the existing runbook has not merely missed mise’s new conveniences; its fresh-machine order cannot install the tracked runtimes as written.

Shell integration has real drift

The interactive block in zshrc is a snapshot of mise activate zsh committed in November 2024. Current mise output now uses add-zsh-hook, separate reason-aware hooks, an initial environment hook, cache state, and a command-not-found recursion guard. Keeping generated implementation code in git guarantees more drift. 

# desired shape, after choosing a portable mise executable
eval "$(mise activate zsh)"

Both current shell files also hardcode /opt/homebrew/bin/mise. Before Homebrew exists they emit startup errors, and the path assumes Apple Silicon. More importantly, macOS path_helper changes precedence in login shells despite the repo’s shim-first rule: 

$ zsh -c 'command -v node; node -v; command -v python3; python3 --version'
~/.local/share/mise/shims/node
v24.13.1
~/.local/share/mise/shims/python3
Python 3.13.14

$ zsh -lc 'command -v node; node -v; command -v python3; python3 --version'
/opt/homebrew/bin/node
v26.4.0
/usr/bin/python3
Python 3.9.6

Homebrew Node cannot simply be uninstalled: it is currently required by agent-browser and mermaid-cli. The correct repair is to reassert mise shim precedence for login shells, then test both zsh -c and zsh -lc in verification.

Why the Brewfile should stay—for now

mise’s new built-in brew manager is substantial: it can pour checksum-verified bottles into the canonical Homebrew prefix, coexist with a real Homebrew installation, and manage core formulae, app/binary/simple-pkg casks, and taps that publish API metadata. But it is not a lossless Brewfile interpreter.

  • Our three font casks use Homebrew’s font artifact, which mise does not support.
  • Cask import and prune are not implemented; formula-only prune can remove formulae installed by real Homebrew.
  • brew services, some cask hooks/artifacts, formula options, and Intel macOS are outside the current implementation.
  • The zimakki/tap Inkwell cask currently has no api/cask/inkwell.json, which mise requires for third-party tap installs.

A hybrid split—some packages in TOML, others in the Brewfile—would create two package sources of truth. Keeping the curated Brewfile behind an idempotent hook preserves its coverage and comments while still allowing mise to orchestrate the overall run.

Recommended path

  1. Repair the current bootstrap order first. Explicitly bootstrap Homebrew/mise, expose the tracked mise config before mise install, export Erlang build settings before tool install, and verify login-shell precedence.
  2. Upgrade mise to 2026.7.4 or newer. The installed 2026.7.1 predates stable bootstrap. Add min_version so the repo cannot silently run against the old CLI surface.
  3. Create one discoverable repo-level mise config. The current custom filename is invisible on a fresh checkout. Design the global self-link carefully because replacing the active global config affects later invocations, not the already-running one.
  4. Pilot declarative dotfiles and defaults. Move 19 static links (including the login-shell zprofile fix) and 12 supported defaults, preview with --dry-run, and add bootstrap status --missing to verification.
  5. Keep deliberate escape hatches small. Retain the Brewfile, the dynamic Lazygit destination, skill synchronization, the -currentHost battery preference, and GUI/credential follow-ups as narrowly scoped scripts or hooks.
  6. Revisit package migration later. Do it only when the font casks and Inkwell tap are representable and a dry-run proves the translated inventory is complete.
