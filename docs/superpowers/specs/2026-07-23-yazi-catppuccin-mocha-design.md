# Yazi Catppuccin Mocha Theme Design

- Date: 2026-07-23
- Status: Approved

## Context

The workstation uses Catppuccin Mocha with Mauve accents across Ghostty,
Starship, Atuin, Television, Hunk, Bat, Lazygit, Warp, and Zsh. Yazi is
installed through Homebrew and its configuration is statically linked from
`config/yazi/`, but it currently uses Yazi's default colors.

Yazi 26.5.6 supports installable flavors through `ya pkg`. The official
`yazi-rs/flavors` repository includes a Catppuccin Mocha flavor with its own
Yazi styles, icons, and matching syntax-highlighting theme.

## Goals

- Make Yazi consistently use Catppuccin Mocha.
- Use the official Yazi flavor rather than recreating the palette locally.
- Keep the setup reproducible through the dotfiles checkout.
- Preserve Yazi's existing session configuration and shell wrapper behavior.
- Keep future flavor updates explicit and reviewable.

## Non-goals

- Automatic light/dark switching or a Catppuccin Latte installation.
- Custom color overrides on top of the official Mocha flavor.
- Changes to Yazi navigation, key bindings, previews, or file operations.
- A general-purpose package installation framework for Yazi plugins.

## Considered Approaches

### 1. Track the official locked flavor and its installed assets

Use `ya pkg add yazi-rs/flavors:catppuccin-mocha`, retain the generated
`package.toml`, and track the complete installed flavor directory alongside the
app configuration.

This is the selected approach. It provides the complete official presentation,
keeps the package revision visible, and makes a fresh dotfiles link immediately
usable without adding a network-dependent bootstrap step.

### 2. Track only the package lock and install during bootstrap

This would keep the repository smaller, but it would add `ya pkg install` and
its network failure modes to the bootstrap exception workflow. The flavor is a
small, intentional app theme, so that complexity is not justified.

### 3. Hand-author `theme.toml`

This would avoid an upstream package but duplicate the palette and Yazi-specific
style decisions. It would also omit or require separately maintaining the
matching preview syntax theme.

## Design

### Managed files

The Yazi configuration will contain:

- `init.lua`, unchanged, for cross-instance yank synchronization.
- `theme.toml`, containing only a `[flavor]` table that selects
  `catppuccin-mocha` for dark mode.
- `package.toml`, generated by `ya pkg`, locking the flavor's upstream revision
  and content hash.
- `flavors/catppuccin-mocha.yazi/`, the complete package-managed flavor,
  including `flavor.toml`, `tmtheme.xml`, its licenses, README, and preview.

No custom style overrides will be added. The package-managed flavor directory
must not be edited directly because local changes can invalidate its recorded
hash and interfere with upgrades.

### Configuration flow

`mise bootstrap` already links the entire `config/yazi/` directory to
`~/.config/yazi`. Yazi reads `theme.toml`, selects the tracked
`catppuccin-mocha` flavor, and uses the flavor's bundled `tmtheme.xml` for code
preview highlighting. No new bootstrap hook or exception script is needed.

On future updates, `ya pkg upgrade yazi-rs/flavors:catppuccin-mocha` will update
both the lock metadata and installed flavor files. Those changes should be
reviewed together before committing.

### Failure handling

- A failed package download must leave the existing Yazi configuration usable;
  partial flavor changes will be inspected before retrying.
- A missing or mismatched flavor will be diagnosed with `ya pkg list` and
  Yazi's debug output.
- `ya` and `yazi` must remain on the same version, as provided by the Homebrew
  formula.

## Verification

- Parse `theme.toml` and `package.toml` as TOML.
- Assert that the selected dark flavor is `catppuccin-mocha`.
- Assert that the lock entry and required flavor assets are present.
- Confirm `ya pkg list` reports the flavor.
- Launch Yazi in a pseudo-terminal and confirm it exits cleanly with the managed
  configuration.
- Run `scripts/ci_checks.sh`, `scripts/bootstrap/preflight.zsh`, and
  `mise bootstrap --dry-run`.
